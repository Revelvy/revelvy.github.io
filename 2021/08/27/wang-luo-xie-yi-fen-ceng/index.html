网络协议分层

通信协议

⭐ 作用：让不同网络架构、不同操作系统的计算机之间互相通信。

OSI

七层，物数网传会表应。

TCP/IP

四层，物（物数）网、传、应（会表应）。

TCP/IP
应用层 HTTP HTTPS FTP Telnet
传输层 TCP UDP
网络层 IP ARP 路由
链路层 以太网 网桥

物理层

传送的数据单位：⽐特。

作⽤：实现相邻计算机节点之间⽐特流的透明传送，尽可能屏蔽掉具体传输介质和物理设备的差异。使数据链路层不必考虑⽹络的具体传输介质是什么。 “透明传送⽐特流”表示经实际电路传送后的⽐特流没有发⽣变化，对传送的⽐特流来说，这个电路好像是看不⻅的。

数据链路层

两台主机之间的数据传输，总是在⼀段⼀段的链路上传输的，这就需要使⽤专⻔的链路层的协议。 在两个相邻节点之间传送数据时，数据链路层将⽹络层交下来的 IP 数据报组装成帧，在两个相邻节点间的链路上传送帧。每⼀帧包括数据和必要的控制信息（如同步信息，地址信息，差错控制等）。
在接收数据时，控制信息使接收端能够知道⼀个帧从哪个⽐特开始和到哪个⽐特结束。这样，数据链路层在收到⼀个帧后，就可从中提出数据部分，上交给⽹络层。
控制信息还使接收端能够检测到所收到的帧中有误差错。如果发现差错，数据链路层就简单地丢弃这个出了差错的帧，以避免继续在⽹络中传送下去⽩⽩浪费⽹络资源。如果需要改正数据在链 路层传输时出现差错（这就是说，数据链路层不仅要检错，⽽且还要纠错），那么就要采⽤可靠性传输协议来纠正出现的差错。这种⽅法会使链路层的协议复杂些。

网络层

在计算机⽹络中进⾏通信的两个计算机之间可能会经过很多个数据链路，也可能还要经过很多通信⼦⽹。⽹络层的任务就是选择合适的网间路由和交换结点， 确保数据及时传送。 在发送数据时，⽹络层把传输层产⽣的报⽂段或⽤户数据报封装成分组和包进⾏传送。在 TCP/IP 体系结构 中，由于⽹络层使⽤ IP 协议，因此分组也叫 IP 数据报 。
这⾥要注意：不要把运输层的“⽤户数据报 UDP ”和⽹络层的“ IP 数据报”弄混。另外，⽆论是哪⼀层的数据单元，都可笼统地⽤“分组”来表示。这⾥强调指出，⽹络层中的“⽹络”⼆字已经不是我们通常谈到的具体⽹络，⽽是指计算机⽹络体系结构模型中第三层的名称。互联⽹是由⼤量的异构（heterogeneous）⽹络通过路由器（router）相互连接起来的。互联⽹使⽤的⽹络层协议是⽆连接的⽹际协议（Intert Protocol）和许多路由选择协议，因此互联⽹的⽹络层也叫做⽹际层或IP层。

IP

Internet Protocol
	作用：标识网络中的主机。
	组成：网络位 + 主机位
	表示：点分十进制
	类型：
    	 IPv4： 4字节、8 8 8 8        [0，255]   255^4
    	 IPv6：16字节、128 128 128... [0,65535] 65535^8
	理解：公司
    分类：
        A   1   ~ 127   网+主+主+主
        B   128 ~ 191   网+网+主+主
        C   192 ~ 223   网+网+网+主
        D   224 ~ 239   组播(多播)
        E   240 ~ 254   科研
公有IP、私有IP
    公有IP：公网地址，通过它直接访问因特网（广域网） 
    私有IP：专网地址，免费为企业内部使用，（局域网），出了所在局域网是无法访问因特网的。 
    私有IP分类：
        A	10.0.0.1    ~ 10.255.255.255
        B	172.16.0.1  ~ 172.31.255.254
        C	192.168.0.1 ~ 192.168.255.254
子网掩码 NETMASK
	长度：32个二进制位
	特点：不能单独存在，必须结合IP地址一起使用
	作用：通过与运算，区分网络位和主机位，判断两台计算机是否在同一子网。当一台计算机访问其他IP地址时，通过比较本机与目标主机的网络位来决定是否需要找网关。
	分类：
        A  255.0.0.0      （1网3主）
        B  255.255.0.0    （2网2主）
        C  255.255.255.0  （3网1主）
网关 GATEWAY
	概念：一个网络连接到另一个网络的“关口”。
	网关设备：连接两个不同网络的设备。
	网关地址：网关设备的IP。
	作用：实现两个网络间的通讯和控制。
域名服务器 DNS
	作用：解析域名（域名和IP之间的解析）

Port

端口，每一个应用都会监听一或多个端口，相同的应用使用这个端口发送程序，但一个端口不能被两个应用同时使用。
一个端口占2字节[0·65535]，个人使用端口最好1w以上。
理解：部门

架构

UDP

概念：User Data Protocol，用户数据报协议（会丢包）
特点：无连接（效率高但安全性低）。
场景：用于对实时性和高速传输要求比较高的通信或广播通信场合。如：即时通信（视频），实时战斗游戏（TCP的特性，一旦丢包就会重发，阻塞住后续的数据包，会有延迟）。
优点：速度快、资源消耗小。
缺点：不可靠性（传输前不建立连接，对接收的数据也不发送确认信号，发送端不知道数据是否会正确接收，不重发）

TCP

概念：Transmission Control Protocol，传输控制协议（不丢包）
特点：有连接，通过3次握手建立连接，数据传输完成时，通过4次挥手断开连接（效率低但安全性高）。
使用场景：用于对可靠性要求比较高的通信场合。如：电子邮件、微信发信息（不能丢包所以TCP）
优点：可靠性：TCP 在数据传递时，有确认、窗口、重传、阻塞等控制机制，能保证数据正确性。
缺点：速度慢，资源消耗大。
3次握手：
    1.客户端向服务端发送一个带有SYN的连接请求。
    2.服务端收到连接请求并确认可以连接之后，客户端回复一个带有SYN和ACK的确认信息。
    3.客户端收到确认请求后，给服务端回复带有 ACK 的确认信息。
    至此三次握手过程结束，客户端开始TCP数据传输。
4次挥手：
    由于 TCP 连接是可以双向通信的（全双工），因此每个方向都必须单独进行关闭
    客户端和服务端都可以先开始断开连接
    1	客户端给服务端发送带有FIN的断连请求
    2	服务端收到断连请求并确定可以断连之后，回复带有ACK的确认信息
    3	服务端回复带有FIN的断连的请求
    4	客户端回复带有ACK的断连确认信息
1、ACK 是TCP报头的控制位之一，对数据进行确认。确认由目的端发出， 用它来告诉发送端这个序列号之前的数据段都收到了。 比如确认号为X，则表示前X-1个数据段都收到了，只有当ACK=1时,确认号才有效，当ACK=0时，确认号无效，这时会要求重传数据，保证数据的完整性。
2、SYN 同步序列号，TCP建立连接时将这个位置1。
3、FIN 发送端完成发送任务位，当TCP完成数据传输需要断开时,，提出断开连接的一方将这位置1。

TCP为什么要3次握手？而不是2次？4次？
    TCP是传输控制协议，其优点是具有可靠性，既要保证数据可靠传输，又要提高数据传输效率，用三次恰恰可以满足以上两方面的需求！
	防止服务器接收到因网络延迟导致失效的客户端连接请求，错误的建立连接，让服务器以为传输连接已建立，就一直等客户端发送数据，浪费服务器资源。（排除2次）
	3次的话，此时客户端不会再给服务器发送确认，服务器由于收不到确认，就不会建立连接。

HTTP

概念
    Hypertext Transfer Protocol 超文本传输协议，是客户端请求和响应的标准协议。
    规定了浏览器和服务器之间互相通信的规则（内容格式）。
    客户端发给服务器的格式叫"请求协议"；服务器发给客户端的格式叫"响应协议"。
特点
    1. 明文传输，端口80
    2. 支持客户端/服务端模式。
    3. 连接简单快速：客户向服务器请求服务时，只需传送请求方法和路径，HTTP服务器程序规模小，通信速度很快。
    4. 灵活：允许传输任意类型的数据对象，传输的类型由Content-Type加以标记。
    5. 无连接：指每次连接只处理一个请求。服务器处理完请求并收到客户的应答后就断开连接，节省传输时间（为弥补不足，产生了两种记录http状态的技术，Cookie 和 Session ）。HTTP1.1 版本后支持可持续连接。通过这种连接,就有可能在建立一个 TCP 连接后,发送请求并得到回应，然后发送更多的请求并得到更多的回应.通过把建立和释放 TCP 连接的开销分摊到多个请求上，则对于每个请求而言，由于 TCP 而造成的相对开销被大大地降低了。而且，还可以发送流水线请求,也就是说在发送请求 1 之后的回应到来之前就可以发送请求 2.也可以认为，一次连接发送多个请求，由客户机确认是否关闭连接，而服务器会认为这些请求分别来自不同的客户端。
    6. 无状态：指对事务处理没有记忆能力，意味着如果后续处理需要前面的信息，则必须重传，这样可能导致每次连接传送 的数据量增大。另一方面，在服务器不需要先前信息时它的应答就较快。
HTTP URL
	http://host[:port]/[abs_path]
	http://IP或域名:端口/资源路径
HTTP 请求
	组成：请求行（第一行）、请求头（所有的键值对）、请求正文（get请求没有）
	请求行格式：Method Request-URI HTTP-Version CRLF
            Method 请求方法  Request-URI 统一资源标识符
            HTTP-Version HTTP 协议版本  CRLF 回车和换行
    请求头：Request Headers 客户端向服务器端传递请求的附加信息以及客户端自身的信息
    	    Referer：该请求头指明请求从哪里来 。
HTTP 响应
	组成：响应行、响应头、响应正文
    响应头：Response Headers 服务器传递不能放在状态行中的附加响应信息、关于服务器的信息、对Request-URI所标识的资源进行下一步访问的信息。
    Location 响应报头域：重定向接受者到一个新的位置（常用于更换域名）。
    	response.sendRedirect("http://www.baidu.com");
	Refresh：自动跳转（单位是秒），可以在页面通过meta标签实现，也可在后台实现。
    	//每3秒刷新一次当前页面
    	<meta http-equiv="refresh" content="3;url=http://www.baidu.com">
    	//3秒后跳转到百度
    	<meta http-equiv="refresh" content="3;url=http://www.baidu.com">
